# Repository Guidelines

## Project Structure & Module Organization
- `contracts/` – Solidity sources (currently `Storage.sol`).
- `scripts/` – operational scripts such as `deploy.js` for deployments.
- `hardhat.config.ts` – Hardhat configuration (Solidity 0.8.28).
- `artifacts/`, `cache/` – build outputs generated by Hardhat.
- Add new tests under `test/` (create the folder if absent); mirror contract names for clarity, e.g., `test/storage.ts`.

## Build, Test, and Development Commands
- Install deps: `npm install`.
- Compile contracts: `npx hardhat compile`.
- Run tests: `npx hardhat test` (use the Hardhat Network by default).
- Local node for manual testing: `npx hardhat node` in one shell, then `npx hardhat run scripts/deploy.js --network localhost` in another.
- Clean build artifacts: `npx hardhat clean`.

## Coding Style & Naming Conventions
- Solidity: 4-space indentation, PascalCase for contract names, camelCase for functions/variables, and prefix private/internal vars with `_` when helpful for clarity.
- JavaScript/TypeScript: 2-space indentation; prefer ES modules (`type`: `module` set in `package.json`).
- Filenames: match contract or script purpose (`Storage.sol`, `deploy.js`). Tests should mirror contract names (`storage.ts`/`storage.test.ts`).
- Run formatters/linters if added; keep imports ordered and avoid unused variables (enable `eslint`/`prettier` in future changes).

## Testing Guidelines
- Use Hardhat’s built-in test runner with Mocha/Chai (default in Hardhat). Place specs under `test/`.
- Name tests descriptively: `should set number to provided value`.
- Aim for coverage of state transitions, access control, and revert paths. Add edge cases around zero values and large integers for storage.
- Prefer deterministic tests; avoid time-dependent logic unless mocked.

## Commit & Pull Request Guidelines
- Commit messages: present-tense, concise summary (e.g., `Add storage setter test`); scope the change to one logical concern.
- Include relevant context in PRs: purpose, key changes, how to run tests, and any deployment steps.
- Link related issues if tracking is used; add screenshots only when UI or console output is meaningful.
- Before opening a PR: run `npx hardhat compile` and `npx hardhat test`; ensure scripts and configs still work on a fresh install.

## Security & Configuration Tips
- Never commit private keys or `.env` files; use environment variables and `.gitignore`.
- Review external dependencies before adding them; prefer audited libraries.
- When deploying, confirm network and gas settings explicitly; keep deployment scripts idempotent where possible.
